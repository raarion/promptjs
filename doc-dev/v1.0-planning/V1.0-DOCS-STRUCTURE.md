# PromptJS — v1.0.0 Documentation Structure Plan

> Rencana restrukturisasi dokumentasi untuk v1.0.0.
> README.md akan diringkas, konten dipindahkan ke `docs/` yang terstruktur.
> `doc-dev/` tetap ada untuk dokumentasi pengembangan internal.

---

## 1. FILOSOFI

**Masalah saat ini:**
- Positioning PromptJS sudah jauh berada di atas yang disuguhkan dan dijelaskan dalam konten README.md
- README.md (790 baris) terlalu panjang dan mencampur overview, tutorial, reference, architecture, dan project map dalam satu file.
- Pengguna baru akan kesulitan menemukan apa yang mereka butuhkan.

**Solusi:** 
- Menperjelas dan mengupdate dengan upgrade position PromptJS sera Framing yang lebih relevan dengan keadaan realtime secara aktual dan faktual
- Mengganti "A mini-DSL template engine that compiles to vanilla JS with zero dependencies." menjadi "A bilingual declarative frontend language that compiles to vanilla JavaScript, with built-in reactivity, components, routing, auth, plugins, and deployment adapters." atau rangkaian kalimat lain yang lebih sesuai.
- Ringkas README.md menjadi landing page (~150 baris) yang mengarahkan ke `docs/` yang terstruktur per topik. Setiap file di `docs/` fokus pada satu subjek dan dapat ditaut dari mana saja.

**Prinsip:**
- README.md = etalase, bukan ensiklopedia
- docs/ = dokumentasi pengguna (user-facing)
- doc-dev/ = dokumentasi pengembang (internal, tetap ada)

## 5. RELASI docs/ ↔ doc-dev/

`docs/` dan `doc-dev/` berbeda secara fundamental:

| Aspek | `docs/` | `doc-dev/` |
|-------|---------|------------|
| Audiens | Pengguna PromptJS | Developer/contributor PromptJS |
| Bahasa | Bilingual (ID+EN) | Campuran (kebanyakan EN) |
| Kelengkapan | Lengkap, rapi, bisa dipublikasikan | Working docs, bisa rough |
| Versi | Selalu terkini (v1.0+) | Termasuk arsip historis (v0.x) |
| Git | Di-commit ke main | Di-commit ke main |

**Alur kerja:**
1. Fitur baru diimplementasi → catat di `doc-dev/` (HANDOFF, planning)
2. Fitur sudah stabil → tulis dokumentasi user di `docs/`
3. `doc-dev/` tetap sebagai sumber kebenaran untuk developer internal

## 7. KONSIDERASI

### Bilingual
Semua docs/ harus bilingual (ID+EN) mengikuti prinsip ⑥ PromptJS.
Pendekatan: paragraf Indonesia diikuti terjemahan English, atau
heading bilingual: `## Instalasi / Installation`.

### Navigasi
Setiap file docs/ harus punya:
1. Header dengan breadcrumb (mis. `> docs/language/ > Auth`)
2. Link "← Kembali" ke parent
3. Link "Selanjutnya →" ke halaman berikutnya (untuk tutorial)

### Diagram
Pipeline dan arsitektur sebaiknya punya diagram visual (Mermaid atau SVG)
di `assets/diagrams/` yang di-embed di `docs/project/compiler-pipeline.md`
dan `docs/project/architecture.md`.

### GitHub Pages
Jika docs/ ingin di-deploy sebagai website:
- Gunakan Jekyll atau MkDocs di root
- Atau deploy folder docs/ sebagai static site
- README.md tetap ada di root untuk GitHub landing
